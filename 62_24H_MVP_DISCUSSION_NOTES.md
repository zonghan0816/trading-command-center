## ✅ 已確認項目

### 1. 內容原則：事實基底 + 活潑風格分離

**確認時間**：2026-05-30 對話

**規則**：
- 語氣可以詼諧、諷刺、批評現象、嘲笑荒謬
- 內容只能引用 topic 跟 keywords 裡實際出現的事實
- ❌ 不替任何政黨/政府/公司/個人下道德判斷
- ❌ 不添加新聞沒提到的指控或陰謀論
- ❌ 不說「政府就是XX」「OO一定收錢」這種臆測
- ✅ 可以諷刺「結構」、「現象」、「規律」、不諷刺特定人
- ✅ 觀察、調侃、反諷都歡迎、但事實基底要乾淨

**Tone 調整對照**：

| Tone | 之前（容易跑偏）| 調整後 |
|---|---|---|
| critical | 「政府爛、政策失敗」 | 「這結構年年炸、修了又修還是漏水」 |
| mocking | 「賴清德這次又踩雷」 | 「政策上路一週就翻車、紀錄保持中」 |
| sarcastic | 「不意外、執政黨從不認錯」 | 「不意外、十年前就這樣演、十年後還是這樣」 |

**落腳處**：`server.py` `_build_prompt()` 加 ~20 行規則 + 修 8 種 tone 的 structure 描述
**優先序**：⭐⭐⭐⭐⭐（24H 開放前必做、法律風險最高）
**改動規模**：5 分鐘 prompt 改動

---

### 2. 新聞來源現況：單源 Google News RSS

**確認時間**：2026-05-30 對話

**現況**：
- 唯一來源：`https://news.google.com/rss?hl=zh-TW&gl=TW&ceid=TW:zh-Hant`
- 每 10 分鐘抓 15 條
- 完全免費、不需 API key

**限制**：
- 單源觀點偏 Google 演算法
- 國際 + 國內混雜
- 沒分類、沒在地度評分

**24H MVP 前需評估加多源**：
- 中央社、自由、聯合、ETtoday RSS
- Yahoo 奇摩新聞 RSS
- ⚠️ PTT 不要爬（會被擋）

**落腳處**：News Curation 中間層（24H MVP Step 2 的一部分）
**優先序**：⭐⭐⭐（24H 上線後可漸進加）
**改動規模**：每多 1 個源 +20 行

---

### 3. 抓新聞成本：完全零成本

**確認時間**：2026-05-30 對話

- Google News RSS = 免費、不限流（合理使用下）
- 多源仍 0 成本（都是免費 RSS）
- 只有 Claude curation（去重/分類）會花錢、但 ~NT$72-360/月、極低

---

### 4. 不能用 Claude.ai / ChatGPT Plus 跑 TDT

**確認時間**：2026-05-30 對話

**原因**：
1. 網頁版沒有 API、無法自動化呼叫
2. 用 Selenium 自動化 = 違反 Anthropic ToS、會封帳號
3. 訂閱有每日訊息上限、24H = 1440 次撐不住

**結論**：必須用 Anthropic API、無法用網頁版訂閱省錢

## 🎯 ★ 產品定位重大澄清（2026-05-30 / 31）

**使用者明確澄清真實目標**：

> 「我其實不是做新聞直播、我要做的是**新聞聊天、討論、批評、嘲諷、詼諧**。
> 我只要做到『**像** 24H AI 直播』、兩個 AI 永遠 live 聊天。
> 直白講就是做一個『**假的** 24H AI 直播』。
> 要說假也沒有啊、確實是 AI 生成的、新聞是真的。」

### 產品本質

| 之前誤解 | 真實定位 |
|---|---|
| 24H 新聞台 / 即時評論頻道 | 24H AI 角色聊新聞**長時間表演** |
| 重視「新聞即時性」 | 重視「角色一致感、不停聊天的氛圍」 |
| 需要 live 緊張感 | 只需要「看起來像 live」 |
| 像 ETtoday 雲端 | 像「預錄廣播 24H 循環」|

### 為什麼「假」不算假

- AI 生成內容 = 真的（Claude）
- 新聞 = 真的（Google News）
- 角色設定 = 真的（阿明小美一致）
- 只有「即時性」是錯覺、但**「報紙印好上市」本來就有時差**、沒人說那是假報紙

### 對成本估算的衝擊

| 模式 | 之前估算 | 真實需求 |
|---|---|---|
| 24/7 全 live + prefetch | NT$57,000/月 | 不需要 |
| **混合模式（95% batch + 5% live）** | — | **NT$100-150/月** |
| 純 batch 模式 | — | NT$450-900/月 |
| 加 Anthropic Batch API 折扣 50% | — | 再砍一半 |

→ **使用者一句澄清、月成本從 NT$57k 變 NT$100-150**

### 對既有架構的衝擊

| 機制 | 之前考量 | 真實需求下 |
|---|---|---|
| Step 6.5 prefetch | 路徑 3 要砍 / 路徑 2 保留 | **完全砍掉**、batch 模式不需要 |
| 即時 `/api/chat` | 主流程 | 5% 用量、僅熱門事件首次 |
| Topic 顯示 | LED 中央 | **UI 拿掉**、後端保留 |
| 反重複機制 | 跨輪比較 | 改成 batch 內保證不重複 |
| Memory Recycle | 路徑 2 補充 | **變成主架構**、不是補充 |
| News Curation | 加值 | **核心**、batch 之前必跑 |

→ 這個澄清是**整個 24H MVP 架構的真正北極星**、之前我跟 GPT 的策略討論都要在這個前提下重新校準。

## 🟡 討論中項目

### 5. Batch 預生成架構（使用者構想 - 已升級為「主架構」）

**討論時間**：2026-05-30 對話

**使用者原話**：
> 抓 30-50 則新聞 → 用 AI 處理 → 套 prompt 規則 → 2 位 AI 講出來

**Claude 翻譯**：
- 不是用網頁版（不可行）
- 而是「批次 API 預生成」(pre-generation batch)
- 每幾小時抓新聞 → 一次 API 處理出 100-200 段對白 → 存 pool → 24H 慢慢播

**成本對照**：

| 模式 | 月成本 |
|---|---|
| 現況（即時 + Step 6.5 prefetch）| NT$58,800 |
| Batch 預生成 | **~NT$300-450** |
| **省幅** | **99.4%** |

**Trade-off**：
- ✅ 超省錢
- ✅ 品質可預挑（人工或自動）
- ✅ 不會即時翻車
- ❌ 失去「即時新聞反應」感
- ❌ 觀眾留言互動更難（但 YouTube Chat 本來就延後）
- ❌ 變成「半小時前錄的廣播」感

**混合策略選項**：
- 90% 時間放 batch 預生成
- 10% 時間（黃金時段或重大新聞）即時生成
- 月成本 ~NT$2,400

**工程挑戰**：
- Batch 一次生 200 段、品質可能比一次生 3 段差
- 解法：每段獨立 prompt + 共享 system message + Anthropic batch API（折扣 50%）
- Pool 管理機制（refill / 過期 / topic 換）

**跟既有 roadmap 的位置**：
- 等於 GPT Phase E（生成-播放分離）的核心
- 等於 Claude Memory Recycle 進階版 + Production Pipeline 具體實作

**已決定（2026-05-31 對話）**：

- ✅ **走混合模式**：95% batch + 5% live（熱門新聞首次出現）
- ✅ **Pool 容量**：30 段未播為基線、剩 15 段時觸發 refill
- ✅ **News dedup**：用 URL hash 比對最近 24-48 小時處理過的、不重複進 batch
- ✅ **內容過期**：每段對白標「生成時間 + 新聞時間」、超過 24 小時跳過、選下一段
- ✅ **新聞一天一輪**：一天後新聞跟對白池一起 reset（突發大事可延 2-3 天）
- ✅ **Refill 時機**：聊到剩 15 段時、背景跑 batch（播放期間 generate、無縫切換）

**待決定**：

- 黃金時段定義（哪幾小時要 live、哪些 batch）
- Batch 一次最佳大小（30 條 / 50 條 / 100 條？）
- Live mode 觸發條件（怎麼判定「熱門大事」、自動 vs 手動）

**落腳處**：24H MVP Step 2-3（核心架構轉折）
**優先序**：⭐⭐⭐⭐⭐（這個方向定了、其他都跟著改）
**改動規模**：中-大、server.py 新增 ~200 行 + 前端 OfficeScene 改播放邏輯

### 6. Topic 顯示拿掉 + 後端保留

**確認時間**：2026-05-31 對話

**使用者原話**：
> 「topic 根本不需要顯示、topic 只會讓 AI 一直在同一個話題」

**拆兩個層次**：

| 層次 | 處理 |
|---|---|
| **UI 顯示**（中央 LED）| ✅ **拿掉**「今日話題：XXX」、改顯示品牌字 / 軟性「正在討論：油價」/ 完全 invisible |
| **後端 prompt** | ✅ **保留**、Claude 仍需要知道在聊哪則新聞、不然會空泛 |

**核心觀念**：
- 觀眾不會看到「今日話題」標籤
- 但 AI 仍有具體新聞當素材
- batch 內 30 則新聞輪流、每段對白標 source

**對舊架構衝擊**：

| 機制 | 變化 |
|---|---|
| `state.topic` 欄位 | 後端保留（給 prompt 用）、前端不再 render |
| `state.mode = 'discussion'` | 對 UI 不重要、保留給後端邏輯 |
| LED 切換動畫 | 大幅簡化 / 改成品牌字輪播 |
| `topic_rotate_loop` | 邏輯改：不再「同 topic 5 輪」、而是「pool 內各新聞自由輪播」 |

**落腳處**：UI 層 OfficeScene + 後端 _topic_rotate_loop 邏輯
**優先序**：⭐⭐⭐⭐（架構級調整、跟 batch pool 同期改）
**改動規模**：前端 ~30 行 + 後端 rotate loop 簡化

### 7. 熱門新聞 live 機制

**確認時間**：2026-05-31 對話

**使用者原話**：
> 「抓到 1 則熱門、只有第一次是用即時聊天、之後也是丟進 batch 裡」

**邏輯**：

```
偵測新聞熱度
  ├─ 熱門（前 1-3 件）→ 第一次 live 模式（即時感）→ 生成後也存進 pool
  ├─ 中熱（前 5-10 件）→ 直接進 batch
  └─ 一般 → 直接進 batch
```

**Live 模式佔比**：

```
24 小時內、熱門新聞首次 live ≈ 5% 用量
其餘 95% = batch pool
```

**成本估算**（混合模式）：

```
95% batch：~NT$12-15/天
5% live：~NT$96/月
總計：~NT$100-150/月
```

**待決定**：

- 「熱門」判定標準（Google News trending / 出現頻率 / 觀眾反應？）
- Live 模式什麼時候啟動、什麼時候結束（一輪 vs 多輪？）

**落腳處**：News Curation 中間層 + Live trigger 機制
**優先序**：⭐⭐⭐（24H 上線後再細調、初期可全 batch）
**改動規模**：~50 行 server.py
